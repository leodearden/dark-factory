# Review Briefing Generator Skill — PRD

## Overview

The `/review-briefing` skill creates and maintains the **review briefing** — a versioned, project-specific configuration file that tells the `/review` skill how to evaluate a particular project. It uses an agent team to inspect the codebase, query project memory, and interview the user to produce a briefing covering smoke tests, critical execution paths, architectural invariants, and known gaps.

The briefing is the single most important input to review quality. Without it, reviews devolve into generic static analysis. With a good briefing, reviews can trace real execution paths, run meaningful smoke tests, and distinguish intentional gaps from bugs.

## Motivation

Setting up review context manually is laborious — you need to understand every subproject's entry points, how to run it, what "working" looks like, which paths are critical, and what's intentionally incomplete. This is exactly the kind of task where an agent team adding knowledge extraction and structured interview is worth the cost: the briefing is written once and updated incrementally, but its quality determines whether every subsequent review catches real issues or generates noise.

## Tech Stack

| Layer | Choice | Rationale |
|-------|--------|-----------|
| Primary agent | Opus (max effort) | Architectural understanding, path tracing, judgment |
| Discovery agents | Sonnet (high effort) | Codebase scanning, entry point detection, test enumeration |
| Memory | fused-memory MCP | Prior decisions, known conventions, project context |
| Task management | fused-memory MCP | Awareness of active task tree (what's done, pending, known-gap) |
| Output | YAML file in project repo | Versioned, human-editable, machine-readable |

## Inputs

```
/review-briefing                          # Create or update briefing for current project
/review-briefing --scope fused-memory     # Create/update briefing for one subproject
/review-briefing --validate               # Validate existing briefing against current codebase
/review-briefing --diff                   # Show what's changed since last briefing update
```

## Output

### Primary: `review/briefing.yaml`

The briefing file. Structure:

```yaml

# Generator version: 1.0

project: dark-factory

# Global invariants that apply across all subprojects

  - "All fused-memory MCP calls include project_id and agent_id"

  fused-memory:
    root: fused-memory/
    description: "Unified memory system combining Graphiti and Mem0 behind MCP interface"

    # How to verify the subproject runs at all
    smoke_tests:

        expect: "exit 0"

        expect: "json_field: status = ok"
```

### Secondary: Memory entries

The skill writes observations about the project's reviewability to fused-memory:
- Discovered entry points and their health
- Areas with poor test coverage
- Architectural concerns noted during discovery

## Process

### Step 0: Load existing context

1. Check for existing `review/briefing.yaml` — if present, this is an **update** not a creation
2. Search project memory for:
   - Recent decisions and architectural choices
   - Known conventions and invariants
   - Active task tree (what's done, what's pending, what's blocked)
3. Read CLAUDE.md, DESIGN.md, and any architecture documentation
4. Read PRDs if available (to understand intended behaviour)

### Step 1: Automated discovery (agent team, parallel)

Spawn Sonnet discovery agents in parallel, one per subproject (auto-detected from project structure). Each agent:

1. **Entry point detection**
   - Find `pyproject.toml` scripts/entry points, `main.py`, CLI definitions, server startup
   - Find `Makefile`, `justfile`, `package.json` scripts
   - Identify how the subproject is meant to be launched

2. **Test enumeration**
   - Map test directories: unit, integration, e2e
   - Identify test runner and configuration
   - Note: which areas have integration/e2e tests vs only unit tests?

3. **Dependency mapping**
   - What does this subproject import from other subprojects?
   - What external services does it need (databases, APIs, other processes)?
   - What config/env vars are required?

4. **Public API surface**
   - Enumerate exported functions, classes, endpoints, MCP tools
   - Note which are entry points to critical paths

5. **Stub/placeholder scan**
   - Quick scan for `TODO`, `NotImplementedError`, `pass` bodies, `...` implementations
   - This informs the "known gaps" section

### Step 2: Critical path synthesis (Opus agent)

The Opus agent receives all discovery output and:

1. **Identifies critical paths** — end-to-end execution flows that represent core functionality
   - Traces from entry point to terminal operation
   - Notes every module boundary crossing
   - Identifies where mocking in tests might hide integration issues

2. **Derives invariants** — rules that must hold across the codebase
   - From code patterns (e.g., "every MCP tool calls `validate_project_id()`")
   - From memory (e.g., prior decisions about error handling patterns)
   - From documentation (e.g., CLAUDE.md conventions)

3. **Drafts smoke tests** — minimal commands that verify the software runs
   - Prefers existing test commands if integration/e2e tests exist
   - Falls back to "start the thing and check it responds"
   - Includes setup/teardown for tests that need services running

4. **Classifies known gaps** — cross-references stubs against:
   - Task tree: is there a pending task for this?
   - Memory: was this an intentional deferral?
   - PRD: was this specified as in-scope?

### Step 3: Interactive review with user

Present the draft briefing to the user section by section and ask:

1. **Smoke tests**: "Here's what I think 'working' means for this subproject. What am I missing? What's the thing that would catch a reify-style 'it doesn't load at all' failure?"

2. **Critical paths**: "These are the end-to-end flows I identified. Are there important ones I missed? For each path, is the trace accurate?"

3. **Invariants**: "These are the rules I derived. Any others? Any of these wrong?"

4. **Known gaps**: "These look intentionally incomplete. Anything I'm flagging as a gap that's actually done? Anything I'm treating as done that's actually a known gap?"

5. **Exclusions**: "I plan to skip reviewing these areas. Anything I should add or remove?"

This is a **conversation**, not a one-shot dump. Iterate on each section based on user feedback before moving to the next.

### Step 4: Generate and write briefing

1. Compile the final briefing YAML
2. Write to `review/briefing.yaml`
3. If updating an existing briefing:
   - Preserve human edits (detect and merge, don't overwrite)
   - Show diff of changes
   - Ask user to confirm before writing

### Step 5: Validate (optional, automatic on update)

Run a validation pass to ensure the briefing is consistent with the current codebase:

- Do referenced modules/files exist?
- Do smoke test commands parse correctly?
- Do referenced tasks exist in the task tree?
- Are critical path module references still accurate?

Report any stale references and offer to fix them.

## Update Mode

When a briefing already exists, the skill operates in **update mode**:

1. Load existing briefing
2. Run discovery (Step 1) to detect what's changed since last update
3. Diff new discoveries against existing briefing:
   - New subprojects or modules not covered
   - Removed or renamed files referenced in briefing
   - New entry points or tests
   - Stubs that have been filled in (known gap → no longer a gap)
   - New stubs that weren't there before
4. Present changes to user for review
5. Merge approved changes into existing briefing (preserving human edits)

## Validate Mode (`--validate`)

Quick check without updating:
- All referenced files exist
- Smoke test commands are syntactically valid
- Known gap task references are still in the task tree
- Critical path modules haven't been moved/renamed
- No new subprojects exist without briefing coverage

Output: pass/fail with list of stale or missing references.

## Diff Mode (`--diff`)

Show what's changed in the codebase since the briefing was last updated:
- New files/modules not covered by any critical path
- Removed files that are still referenced
- New entry points not covered by smoke tests
- Stubs filled in (can remove from known_gaps)
- New stubs introduced (candidates for known_gaps)

This helps users decide whether to run a full briefing update.

## Configuration

No additional config beyond the review section already defined for `/review`:

```yaml
review:
  briefing_path: "review/briefing.yaml"    # Where the briefing lives
```

## Non-Goals

- **Auto-generating tests** — the skill identifies what needs testing, not writes the tests
- **Running the review itself** — that's the `/review` skill's job
- **Replacing human architectural knowledge** — the interview step is critical; pure automation produces shallow briefings
- **One-shot generation** — the briefing is a living document; the skill is designed for iterative updates
- **Cross-project briefings** — each project gets its own briefing; no shared/global briefing

## Verification

### The skill works when:
1. Running `/review-briefing` on a new project produces a valid `review/briefing.yaml` with all sections populated
2. Discovery agents correctly identify entry points, tests, dependencies, and stubs
3. Critical path traces follow real code execution, not guessed paths
4. The interactive review step asks meaningful questions and incorporates user feedback
5. Running `/review-briefing` on a project with an existing briefing correctly identifies changes and preserves human edits
6. `--validate` catches stale references (renamed files, removed modules, completed known-gaps)
7. `--diff` correctly identifies new coverage gaps
8. The briefing is valid YAML that the `/review` skill can parse
9. The briefing includes enough detail that an Opus agent can trace critical paths without additional human guidance
10. Known gaps are correctly cross-referenced against the task tree and memory
